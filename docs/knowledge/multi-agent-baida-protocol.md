# 多 agent 白大协作协议

## 1. 适用范围

只要任务涉及选股、买卖、仓位、日报口径、回测、候选池、Formula33、基本面评分、技术量化、板块主线或生产流水线，就默认采用三角色协作。

纯格式、拼写、路径或局部工具修复可以由主 agent 直接完成，但如果改动会影响股票筛选或买卖解释，仍要进入本协议。

## 2. 角色分工

### 主 agent

主 agent 负责主流程运行和交付：

- 先读 `STRATEGY.md`、`docs/README.md` 和相关知识文档；
- 判断任务是否触发白大 agent 与量化 agent；
- 维护生产七步顺序：`formula33 -> sector_stats -> sector_watch -> factor_selection -> fundamental_update -> fundamental_selection -> daily_report`；
- 负责代码修改、命令执行、测试、回测、结果汇总和最终答复；
- 遇到冲突时以已落地策略文档、时点数据规则和可回归验证为准，并把冲突记录下来。

主 agent 不能把研究建议直接写成规则。任何新规则都必须转成可解释、可回测、可时点复现的条件。

### 白大 agent

白大 agent 是业务与交易思想顾问，重点负责“为什么这样选、为什么这样买卖”：

- 调用 IMA 和网页查询，补充业务场景、公司逻辑、产业变化、市场叙事和技术细节；
- 对选股逻辑给出白大思想下的建议：左侧价值、右侧强势、主线板块、量价结构、回调修复、突破确认、风险边界；
- 对买卖逻辑给出建议：建仓触发、加仓条件、止损、分仓止盈、继续观察或剔除原因；
- 明确区分事实、推断和交易假设；
- 输出可以被主 agent 和量化 agent 检查的规则建议，而不是只给主观结论。

白大 agent 的建议必须回到项目口径：

- 不绕过观察日和时点数据；
- 不用未来数据解释历史入选；
- 不把新闻热度、主观叙事或单一技术信号直接当作买入指令；
- 不破坏价值线、正常基本面、Formula33 和板块主线的边界。

### 量化 agent

量化 agent 是纠错与一致性审查者，重点找冲突：

- 检查当前代码哪里不符合白大思想；
- 检查选股模型、排序、过滤、买卖规则和报告文案是否互相冲突；
- 检查是否违反 `STRATEGY.md`、观察日、复权、停牌、数据源、候选池和日报门禁；
- 检查回测是否有幸存者偏差、未来函数、样本泄漏、冻结数据误用或只看总收益的问题；
- 给出最小修复建议和必须补的测试或回归命令。

量化 agent 的输出优先按风险排序：阻断问题、策略冲突、数据问题、测试缺口、可优化项。

## 3. 标准协作顺序

1. 主 agent 明确任务边界，读取现行文档和相关代码。
2. 白大 agent 给出业务场景、技术细节、买卖逻辑和选股逻辑建议。
3. 主 agent 将建议整理成可实现规则，先标出与现行文档的差异。
4. 主 agent 做代码或文档修改。
5. 量化 agent 审查新旧逻辑冲突、未来函数、时点数据、回测口径和测试缺口。
6. 主 agent 根据审查结果修正并运行必要验证。
7. 主 agent 汇总：改了什么、验证了什么、仍有哪些风险。

如果当前环境无法实际创建子 agent，主 agent 也必须按这三个视角自检，并在输出中说明哪些部分由主 agent 代做。

## 4. 白大思想落地口径

项目中“白大思想”不是单一公式，而是一组决策约束：

- 先区分左侧价值、正常基本面、右侧强势和板块主线，不把不同维度混成一个万能分数；
- 左侧重视价值线、质量、增长的真实性和安全边际；
- 右侧重视 Formula33 市场结构、主流板块、量价配合、突破和回调修复；
- 买入前必须能解释：为什么是这只、为什么是现在、错了在哪里止损；
- 卖出和减仓必须能解释：价值证伪、趋势破坏、风险过高、仓位纪律或更优机会；
- 观察名单、交易计划和人工判断只能作为显式结构锚点，不能直接注入候选池；
- 所有自动化结果只服务提醒和研究，不连接券商，不自动下单。

## 5. 工具使用规则

- IMA 和网页查询由白大 agent 优先使用，用于当前业务、行业、公司、技术和买卖逻辑补充。
- 代码搜索、测试、回测、数据库和流水线命令由主 agent 负责。
- 回测和模型审查由量化 agent 提出检查点，主 agent 负责执行验证。
- 当前事实可能变化时必须联网核实，并在结论中区分“已核实事实”和“交易推断”。
- 任何外部信息进入模型前，都必须转成观察日可获得、可复现、可测试的字段或规则。

## 6. 冲突处理

当白大建议、量化审查和现行代码冲突时：

1. 先保留现行生产不变量和 `STRATEGY.md` 口径；
2. 再判断冲突是文档过时、代码偏离、数据不可得，还是建议尚未量化；
3. 能验证的用回测、夹具或固定回归验证；
4. 不能验证的先写成观察项或研究计划，不直接进入正式候选池；
5. 必须说明冲突和处理结论。

## 7. 交付检查清单

涉及策略的任务完成前，主 agent 至少检查：

- 是否读过本协议和现行策略文档；
- 是否从白大视角检查业务场景、技术细节、买卖逻辑和选股逻辑；
- 是否从量化视角检查冲突、未来函数、数据时点和回归风险；
- 是否没有绕过七步生产门禁；
- 是否运行了与改动风险匹配的测试、回测或静态检查；
- 是否把无法自动验证的部分明确标为人工判断或后续研究。
