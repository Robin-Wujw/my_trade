# 21日三浪三趋势与双消息日报设计

## 目标

每日分析只使用“最近21个交易日三浪三命中股票去重数”判断市场宽度和右侧开关，不再使用单日命中数或单日变化。PushPlus 改为两条结构化消息，在保持可读性的同时完整列出两类选股；自托管 GitHub Actions 每天完整执行，不再设置三日门控。

## 当前问题

1. 日报用最新一天的 `count`、`change`、`up_streak` 和 `down_streak` 判断右侧开关，单日波动会主导结论。
2. `formula33_stats` 只有最新一行填写 `window_unique_count`，此前交易日均为空，无法计算滚动趋势。
3. PushPlus 把四个栏目压入一条消息，只详细显示前几只股票，其余股票只剩名称，层次和分析链都不清晰。
4. Workflow 虽然每天收到定时事件，但三日门控只允许每三个 UTC 日完整执行一次，绿色运行记录可能只是门控跳过。
5. 最终日报没有稳定保存最近两个不同交易日的正式名单，因此无法可靠标出新进入、退出和策略分区变化；同日重跑还可能误用当天结果作为对比基线。

## 指标定义

### 21日三浪三去重数

对每个交易日 `d` 定义：

```text
window_unique_count(d)
= 在 [d 前第20个交易日, d] 内至少命中过一次 XG 的股票代码去重数
```

该指标沿用现有“技术去重”口径，不因观察日停牌而删除历史命中。`tradable_unique_count` 继续描述最新观察日可交易集合，但不用于市场趋势和右侧开关。

### 21日滚动趋势

对每个交易日 `d`，取截至 `d` 的最近21个 `window_unique_count` 值，以交易日序号 `0..20` 为横轴做普通最小二乘线性回归，斜率记为 `window_trend_slope(d)`。

- 斜率 `> 0`：该观察日为正趋势日；
- 斜率 `< 0`：该观察日为负趋势日；
- 斜率 `= 0` 或有效样本不足21个：中性，正负连续计数都归零。

从最新观察日向前统计连续正趋势日数和连续负趋势日数，写入 `trend_up_streak`、`trend_down_streak`。为了给最近21个输出日都生成完整趋势，需要一次保留至少61个交易日的 XG 命中：61日原始命中生成41个完整21日去重值，再生成21个完整回归斜率。

### 右侧开关

只使用滚动趋势连续日数：

| 条件 | 结论 |
| --- | --- |
| `trend_down_streak >= 5` | 暂停右侧交易 |
| `trend_down_streak >= 3` | 谨慎或暂停右侧 |
| `trend_up_streak >= 5` | 可以右侧交易 |
| `trend_up_streak >= 3` | 可以谨慎右侧 |
| 当前斜率为正但连续不足3日 | 轻仓观察右侧 |
| 其他情况 | 等待右侧确认 |

日报可以保留行情覆盖、停牌和数据不可用诊断，但不得显示或引用单日 `count`、`change`、旧 `up_streak/down_streak` 作为市场结论依据。底层 CSV 可暂时保留这些兼容字段供诊断和历史回归使用。

## 数据计算与文件职责

### `stock_research/pipelines/formula33.py`

- 将生产运行的内部计算日期扩展到足以生成完整滚动指标的61个交易日，公开 `--lookback 21` 的含义仍是最终统计与报告窗口。
- 从 XG 命中记录计算每个观察日的 `window_unique_count`。
- 对每日21日去重序列计算 `window_trend_slope`、`trend_up_streak`、`trend_down_streak` 和趋势信号。
- 最终统计 CSV 输出最近21个交易日；这21行均具有有效的 `window_unique_count` 和 `window_trend_slope`，不再只填最新一行。
- 最新行继续填写 `tradable_unique_count`、`suspended_count` 和 `unavailable_count`。

纯计算逻辑应进入 `stock_research/strategies/formula33.py` 或另一个同层纯函数，避免把滚动集合和回归规则埋在网络抓取流程中。

### `stock_research/reporting/daily_report.py`

- `render_formula_status` 只渲染21日技术去重数、最新斜率、连续正/负趋势日数、正式可交易数和观察诊断。
- `right_side_conclusion` 接收滚动趋势字段，不再读取单日数量和旧连续涨跌字段。
- 完整本地 HTML 仍保留四个栏目和逐股详细原因。
- PushPlus 改为两个独立 HTML 正文，由结构化生成器控制长度，不对拼好的 HTML 做无语义的中间截断。

### `stock_research/reporting/diff.py`

- 为最终两类选股提供按交易日持久化的快照，状态文件固定为 `var/state/daily_selection_history.json`。
- 每个快照至少保存报告日期、股票代码、名称和 `strategy_part`；同一报告日期重跑时覆盖当天快照，不删除更早交易日快照。
- 比较时固定选择“小于当前报告日期的最近一个快照”，因此同日重跑不会把当天第一次结果当作上一交易日。
- 状态文件保留最近若干个不同交易日快照，以支持重复运行和故障恢复；不从导出文件修改时间推断上一交易日。

差异分类固定为：

| 分类 | 定义 |
| --- | --- |
| 新进入 | 股票代码不在上一交易日两类最终池中，但在当前两类最终池中 |
| 退出 | 股票代码在上一交易日两类最终池中，但不在当前两类最终池中 |
| 分区变化 | 股票代码两个交易日都存在，但 `strategy_part` 在“价值线池”和“正常基本面池”之间变化 |

分区变化不得同时计入新进入和退出。缺少更早快照时显示“首次建立可比较基线”，不得把当前全名单标为新进入。完整 HTML 和两条 PushPlus 都必须列出变化股票的名称、代码及变化方向；无变化时明确显示“无新进入、无退出、无分区变化”。

## PushPlus 双消息结构

每条消息上限设为18,000字符，为普通实名用户20,000字符限制保留安全余量。标题均控制在100字符以内。

### 第一条：`[1/2] 日期｜市场状态与价值线池`

顺序固定为：

1. 一句话总判断；
2. 21日三浪三去重数、斜率、连续趋势日数和右侧开关；
3. 与上一交易日相比的总体新进入、退出和分区变化，并列出股票名称、代码与变化方向；
4. 价值线池整体分层：低于50%、50%至62.5%、达到62.5%、质量分不低于80的数量；
5. 价值线池本身的新进入、退出、转入和转出；
6. 优先关注股票的完整分析；
7. 价值线池全部股票的紧凑表格，至少保留名称、代码、现价/价值线、质量分、右侧位置和风险标签；
8. 价值适用性与左侧风险说明。

### 第二条：`[2/2] 日期｜基本面候选与主线`

顺序固定为：

1. 一句话总判断；
2. 正常基本面池整体分层：50%至62.5%、达到62.5%、质量分不低于80的数量和行业集中度；
3. 正常基本面池本身的新进入、退出、转入和转出；
4. 优先关注股票按“业绩硬条件 → 质量/流动性 → 主流板块 → 长期扣抵 → 右侧位置 → 风险”给出完整分析；
5. 正常基本面池全部股票的紧凑表格，至少保留名称、代码、质量分、主流板块、右侧位置和风险标签；
6. 主流板块按数据新鲜度、3/5/20日趋势、量能、强弱和涨停扩散给出依据；
7. 异常同比、板块过期和数据缺失的统一风险说明。

两条消息都必须先生成、检查长度并确认含有各自完整股票集合，之后才开始发送。若丰富版本超过18,000字符，先压缩措辞和非关键解释，再压缩为逐股关键字段；不得静默删除股票。最紧凑版本仍超限时，本次推送失败并保留完整本地 HTML/CSV，而不是发送被截断的残缺名单。

发送顺序固定为第一条后第二条，分别记录 `PUSH_RESULT_1` 和 `PUSH_RESULT_2`。任一请求失败，日报进程返回失败；日志必须明确失败部分。测试中所有发送均使用替身，不调用真实 PushPlus。

## 文档口径

`STRATEGY.md` 中关于三浪三日报的文字改为：

- 只报告最近21个交易日三浪三命中股票去重数；
- 使用该每日滚动值的21日线性趋势和连续正负趋势日数判断右侧开关；
- 删除“当日命中数”“较前一交易日变化”和用单日连续扩张/收缩作结论的说法。

其他运维或架构文档若存在同类表述，也同步更正。策略仍区分21日技术去重数与最新观察日正式可交易数。

## GitHub Actions

`.github/workflows/stock-selection.yml` 保留每天 `08:30 UTC`（北京时间16:30）的 cron 和手动触发入口，删除 `Three-day schedule gate` 以及所有依赖门控输出的 `if` 条件。每次定时事件都依次执行 checkout、运行时验证、完整测试、缓存准备、全市场选股和 Artifact 上传。

自托管 Windows runner 服务继续使用自动启动。电脑关机或休眠时无法执行；开机后服务会自动上线并接取仍在 GitHub 队列期限内的任务。完整任务执行期间电脑必须保持联网和不休眠。

## 测试策略

1. 纯函数测试：构造跨61个交易日的 XG 命中，验证每日21日去重数、21日回归斜率和正/负/中性连续计数。
2. 右侧开关测试：覆盖正负趋势连续0、1、3、5日及样本不足。
3. 日报渲染测试：断言正文包含21日去重数、斜率和趋势结论，且不包含“当日XG”“较前一交易日”等单日措辞。
4. 双消息测试：断言标题顺序、两个正文各自栏目、85只样例全集、每条长度不超过上限，以及任一发送失败时进程失败。
5. 差异测试：覆盖新进入、退出、双向分区变化、无变化、首次基线和同日重跑，断言分区变化不会重复计入新增/退出。
6. Workflow 静态测试：断言 cron 保留、三日门控不存在、主要步骤不再受门控条件控制。
7. 全量验证：运行 `compileall`、全部 pytest、六份历史输出基线；使用现有本地数据生成 `--no-push` 报告并解析 HTML、股票覆盖、差异覆盖和字符长度。
8. 实际验收：确认本地报告无误后发送一次两部分 PushPlus，分别记录接口结果；再检查 runner 在线、自动启动、标签与 Action 配置一致。

## 非目标

- 不修改三浪三 XG 公式本身。
- 不用单日数量补充或覆盖21日趋势结论。
- 不改变基本价值线、基本面筛选和板块评分条件。
- 不把完整本地 HTML 上传到公开地址。
- 不在本次变更中加入交易下单或真实持仓逻辑。
