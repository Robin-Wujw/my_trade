# 数据基础与工程分层重构设计

- 状态：已由用户确认
- 日期：2026-07-01
- 范围：DuckDB 单一事实源、数据时点、历史财务仓库、三浪三市值过滤、工程分层、运行一致性与测试
- 非范围：持仓台账、止盈止损规则、券商接入和自动下单

## 1. 背景

当前项目是一套沪深 A 股收盘后全市场研究与选股流水线。它已经覆盖行情缓存、三浪三市场结构、板块主线、技术因子、基本面截面、价值线计算、日报和 PushPlus 推送，但仍以顶层脚本和文件缓存为主要边界。

现状中最影响后续扩展的问题是：不同结果可能按文件修改时间被拼入同一报告；历史分析没有统一观察日上下文；财务缓存主要服务当前指标，不能完整重建任意历史观察日；`factorStock.py` 等文件职责过多；三浪三生产参数暂时跳过了 100 亿元市值条件。

## 2. 已确认的业务决定

1. 每次分析必须进行数据日期校验，禁止跨日期结果静默组合。
2. 行情和所有派生技术指标必须按观察日截断。
3. 三浪三先计算技术命中，再用观察日可见的总市值筛选 100 亿元以上股票。
4. 完整保存全市场历史资产负债表、利润表和现金流量表，而不只保存当前策略字段。
5. 财报原始版和修订版同时保留；历史查询只能看到观察日当时已经披露的版本。
6. 采用渐进式分层重构，保留现有脚本作为兼容入口，避免一次性重写。
7. 止盈止损延后到独立项目；将来由用户提供持仓数据。
8. DuckDB 是行情、财务、板块、指标、运行记录和选股结果的唯一事实源；Excel/CSV 不再是生产产物或模块接口，只允许按需导出。

## 3. 方案比较

### 3.1 渐进式分层重构（采用）

在不改变当前入口命令的前提下建立包结构、数据仓库和统一运行上下文，再逐步迁移现有逻辑。优点是可以逐阶段回归、并行验证和快速回退；代价是迁移期会暂时存在新旧两套内部路径。

### 3.2 原地修补（不采用）

直接修改现有脚本可以最快解决个别日期问题，但大文件和隐式文件依赖会继续扩散，完整财报版本仓库也没有清晰归属。

### 3.3 全量重写（不采用）

重写可以立即得到整洁结构，但当前策略规则多且已经积累大量本地数据，一次切换容易引入策略口径回归、缓存迁移错误和生产中断。

## 4. 目标架构

```text
my_trade/
├─ src/my_trade/
│  ├─ domain/       # 股票、日期、财报版本、运行清单等领域模型
│  ├─ data/         # AkShare、Baostock、本地文件等数据源适配器
│  ├─ storage/      # DuckDB schema、事务、仓储与时点查询
│  ├─ market/       # 行情、股票池、板块、市值访问服务
│  ├─ indicators/   # 三浪三、波段、均线、均量等纯计算
│  ├─ strategies/   # 价值线、基本面、主线和筛选规则
│  ├─ pipelines/    # 历史回填、每日分析、运行清单与门控
│  └─ reporting/    # CSV、HTML、PushPlus和诊断输出
├─ tests/
│  ├─ unit/
│  ├─ integration/
│  └─ regression/
├─ docs/
└─ 现有顶层脚本          # 迁移期兼容入口
```

模块依赖只允许从外向内：入口和流水线可以调用策略、指标与数据服务；策略只能依赖领域模型和明确接口；指标是无网络、无文件副作用的纯计算；数据源适配器不能反向依赖策略。

## 5. 运行上下文与产物一致性

每次正式运行创建一个不可变 `RunContext`，至少包含：

- `run_id`：由观察日和唯一时间标识组成；
- `observation_date`：本次分析允许看到数据的最后自然日；
- `market_cutoff`：不晚于观察日的最后交易日；
- `financial_cutoff`：财报披露与修订允许进入本次查询的最晚时间；
- `report_period`：按观察日可见规则选出的报告期；
- `code_version`：Git 提交或可追踪的代码版本；
- `created_at`：运行创建时间；
- `mode`：生产、回测或离线验证。

每一步产物必须写入同一 `run_id` 对应的数据库记录，并登记到 `ops.runs` 与 `ops.run_steps`。报告生成器只查询该运行的结果，不再分别按修改时间寻找“最新文件”。任何必需结果的 `run_id`、观察日或截止日不一致时，报告门控失败并停止推送。

运行数据统一进入：

```text
.data/my_trade.duckdb
├─ ops.runs / ops.run_steps
├─ derived.market_structure / derived.sector_watch
├─ derived.strategy_results / derived.diagnostics
└─ ops.reports
```

HTML 是面向用户的展示物：正文首先保存在 `ops.reports`，允许将指定报告导出为独立 HTML 供浏览。PushPlus 发送同一记录的受限摘要。Excel/CSV 仅由显式导出命令临时生成，不得成为后续计算输入。

## 6. 完整历史财务仓库

### 6.1 单一数据库与四个 schema

数据库文件固定为 `.data/my_trade.duckdb`。它是唯一事实源，按职责划分：

1. **`raw`**：数据源原始响应和内容哈希，只追加，不覆盖。
2. **`core`**：标准化股票、交易日、日线、板块、股本、财报版本和财务事实。
3. **`derived`**：观察日财务截面、指标、市场结构、板块主线和策略结果。
4. **`ops`**：运行、步骤、抓取任务、覆盖率、诊断、报告和 schema 迁移记录。

核心表结构由 [DuckDB 数据库结构](../../knowledge/database-schema.md) 约束。数据库外只保留备份和用户明确要求的临时导出，不保存会被生产逻辑再次读取的 JSON、CSV 或 Excel 数据副本。

### 6.2 原始财报与版本

`raw.financial_payloads` 完整保存三大报表源响应，至少记录：

- 股票代码；
- 报表类型：资产负债表、利润表、现金流量表；
- 报告期；
- 公告时间 `published_at`；
- 修订公告时间 `revised_at`，原版为空；
- 时点生效时间 `effective_at`；
- 抓取时间 `retrieved_at`；
- 数据源和源记录标识；
- 原始内容哈希 `version_hash`；
- 完整原始载荷 `payload_json`；
- 字段数、完整性状态和错误说明。

相同内容哈希不会重复入库；内容变化必须形成新版本，禁止覆盖旧版本。`core.financial_reports` 保存标准版本元数据，`core.financial_facts` 保存可查询的完整报表字段。

### 6.3 派生指标与结果

策略常用字段进入 `derived.financial_snapshots`，包括每股净资产、扣非 EPS、扣非净利润同比、总股本、总市值、质量分、流动性分和价值线。指标和策略结果分别进入 `derived.indicator_values` 与 `derived.strategy_results`。每条派生记录必须带上输入版本哈希、算法版本、`run_id` 和观察日，从而在算法变化后可从数据库原始载荷重新计算。

### 6.4 写入模型

外部接口可并发抓取，但只有流水线主进程负责批量写数据库。每个工作线程返回标准化批次，由单写入协调器在事务中执行去重、版本登记和任务状态更新。其他进程不得同时以读写模式打开生产数据库；只读分析需在没有生产写事务时进行，或连接一致性备份。

## 7. 历史回填与每日增量

### 7.1 首次回填

1. 冻结本次股票池和回填任务清单。
2. 按股票、报表类型和可用报告期发现历史记录。
3. 按 `AkShare → Baostock → 本地已有数据` 顺序获取；AkShare 成功后不再请求 Baostock。
4. 在单个 DuckDB 事务中保存原始载荷、版本目录和完整财务事实。
5. 对三表配对、报告期连续性、披露时间和字段完整性进行审计。
6. 将成功和失败任务写入 `ops.ingestion_tasks`，允许安全续跑。

“完整历史”定义为：对受支持的沪深 A 股，保存数据源能够提供的全部历史三表版本；源端确实缺失的记录必须登记为明确缺失，不能用当前值回填或伪造历史。

### 7.2 每日增量

每日只检查上次同步后新增的报告期、公告和修订版本。抓取失败的股票进入重试队列；单股失败不终止整个增量任务，但会降低覆盖率并进入诊断报告。正式选股在覆盖门槛不满足时不得发布。

## 8. 严格时点查询

所有生产选股和历史回测必须通过统一 `AsOfQueryService` 获取数据，禁止策略直接读取任意最新文件。

财报版本选择规则：

1. `effective_at <= observation_date`；
2. 在同一股票、报表类型和报告期中选择观察日当时最新生效版本；
3. 数据源按固定优先级决定，同一查询中不得静默混换口径；
4. 返回值携带来源、版本哈希和时点状态。

行情规则：先过滤 `trade_date <= market_cutoff`，再计算波段、均线、均量、扣抵和收益。板块数据必须不晚于观察日且距观察日不超过七天。当前成分不得回填历史板块成分。

## 9. 三浪三市值过滤

三浪三分为两个明确阶段：

1. 对观察日前行情计算 KDJ、WR、RSI、上市天数和连续五日技术条件，得到技术命中集。
2. 仅对技术命中集查询观察日可见总市值，保留严格大于 100 亿元的股票。

市值缺失采取失败关闭策略：不进入正式 XG 结果，同时写入 `derived.diagnostics`。日报同时报告技术命中数、市值有效数、市值过滤后命中数和缺失数，避免把数据缺失误判为策略收缩。

## 10. 兼容迁移顺序

### 阶段一：基础边界

建立包目录、DuckDB schema、`RunContext`、运行记录、日期校验和测试骨架。顶层脚本继续作为入口，结果口径不变。

### 阶段二：财务仓库

实现数据库原始层、版本表、完整历史回填、每日增量和时点查询。旧财务缓存继续服务生产，新数据库后台建立覆盖率。

### 阶段三：策略拆分

把 `factorStock.py` 中的数据访问、指标、估值、评分和输出逐块迁移到新模块；随后迁移三浪三、基本面和日报组装。每次迁移均以旧结果做回归基线。

### 阶段四：生产切换

日报改为只查询指定 `run_id`；启用观察日截断和三浪三市值过滤；停止自动生成 Excel/CSV；新旧流水线并行验证，达到门槛后切换，旧入口保留一个稳定周期再移除内部旧路径。

## 11. 错误处理与可观测性

- 日期冲突、未来数据、必需产物缺失：立即阻止正式报告。
- 单股抓取失败：记录代码、阶段、来源、错误类别、重试次数和下次重试时间。
- 数据源回退：记录首选源失败原因和实际来源，禁止静默回退。
- 财报三表不完整：保留原始数据但标为不完整，不进入依赖完整三表的派生指标。
- 原始载荷哈希与版本记录不一致：完整性审计失败并阻止使用相关版本。
- 每次运行输出覆盖率、缺失数、回退数、版本新增数、修订数、耗时和最终门控状态。

## 12. 测试策略

### 12.1 单元测试

- 观察日之前和之后的 K 线截断；
- 原版、一次修订和多次修订的版本选择；
- 同内容哈希去重和不同内容生成新版本；
- 财报可见日期、市场截止日和板块七日新鲜度；
- 三浪三技术命中后再筛选 100 亿元；
- 市值缺失失败关闭；
- 价值线及核心派生指标的纯函数计算。

### 12.2 集成测试

- 中断后恢复历史回填；
- AkShare 失败后 Baostock 回退并记录来源；
- 同一 `run_id` 的数据库完整流水线；
- 混入其他日期产物时报告门控拒绝；
- 离线模式仅查询 DuckDB 生成日报；
- Excel/CSV 导出结果不得被任何生产步骤读取。

### 12.3 回归测试

选择若干已经存在结果的历史观察日，比较重构前后的候选股票、关键财务字段、波段锚点和评分。差异必须能归因于已批准的新规则，例如未来数据修正或 100 亿元过滤；无法解释的差异阻止切换。

## 13. 覆盖率与验收

迁移期沿用策略规范的最低门槛：行情覆盖率低于 90% 或当前报告期财务覆盖率低于 35% 时禁止发布；任一覆盖率低于 95% 时必须显示警告。正式切换到新数据层前还需要：

- 当前股票池行情覆盖率不低于 95%；
- 当前可见报告期财务覆盖率不低于 95%；
- 所有已抓取版本通过原始载荷与版本哈希一致性审计；
- 指定历史样本的时点测试全部通过；
- 离线完整日报成功；
- 新旧流水线连续多次运行，无不可解释的结果差异；
- 生产日报不存在跨 `run_id` 或跨观察日输入。

## 14. 安全与运维边界

PushPlus token 等凭证继续只从环境变量或被忽略的本地文件读取，不写入数据库、日志或报告。DuckDB 写入使用显式事务和单进程写入协调器；数据库文件不提交 Git，也不作为 GitHub Actions artifact 全量上传。备份通过 DuckDB 的一致性导出或受控检查点生成，不能在活跃写事务中直接复制数据库文件。

## 15. 后续独立项目

本设计刻意不定义持仓和止盈止损。数据基础和工程迁移稳定后，再单独设计持仓输入、策略分层风控、收盘信号、次日开盘回测成交和人工执行提醒。
