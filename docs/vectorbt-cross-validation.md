# vectorbt 双引擎交叉验证

## 最终回测成交明细

每次组合回测都会同时输出：

- `*_events.csv`：包括成交、批次合并和突破确认等全部策略事件；
- `*_trades.csv`：只包括真实发生资金变化的逐笔买卖；
- `*_summary.json`：在总体收益、回撤和最终持仓之外，内嵌完整的 `trade_ledger` 与 `trade_summary`。

逐笔交易记录包含日期、股票代码与名称、买卖方向、成交动作、成交价、数量、成交金额、佣金、印花税、滑点、费用合计、现金变化、买卖理由和结构锚点。卖出记录还包含成本金额、毛盈亏、分摊的买入费用、扣除买卖两端费用后的净盈亏及收益率。未卖出的股票在 `final_positions` 中给出投入金额、期末市值和未实现盈亏。

## 边界

现有策略引擎继续负责：

- 时点化候选池和Formula33状态；
- L/H/P锚点及50%、62.5%、75%价格结构；
- 条件单的日K触发规则；
- 统一候选接口、交易批次和止盈止损状态；
- 普通A股100股一手、`688`科创板200股一手的成交约束。

vectorbt 1.1.0独立负责：

- 共享现金账户；
- 按实际成交数量重放买卖；
- 万0.85且最低5元的佣金；
- 卖出印花税和估算滑点；
- 每日盯市、净值和最大回撤；
- 现金不足造成的拒单或部分成交。

策略成交事件使用整手后的`execution_quantity`和实际`cost_basis`字段，避免使用展示用的两位仓位百分比重建订单。

## 使用

```powershell
python -m pip install -e ".[vectorbt]"
python -m apps.portfolio_backtest `
  --start-date 2026-01-01 `
  --end-date 2026-07-10 `
  --vectorbt-cross-check
```

只有明确冻结本地输入做可重复比较时才加`--no-refresh-inputs`。正常运行仍应先刷新到市场确认的最近完整交易日。

## 2026-07-10冻结数据初次诊断（修复前）

| 执行方案 | 原引擎收益 | vectorbt收益 | 原引擎/BT最大回撤 | 订单问题 | 最大逐日净值差 |
|---|---:|---:|---:|---:|---:|
| 14:55/收盘代理 | 41.921% | 41.949% | -17.312% / -17.257% | 2次部分成交、1次拒单 | 3.536pct |
| 收盘确认后次日开盘 | 34.360% | 33.863% | -14.721% / -13.143% | 3次部分成交 | 1.538pct |

第一轮确认当时引擎的`gross_exposure <= 100%`不是现金约束。它按初始资本的名义比例分配仓位，却没有在决定下一笔仓位时预留佣金、印花税、滑点，也没有使用已实现盈亏更新可用现金。因此：

- 2026-05-12中际旭创加仓只能部分成交；
- 2026-05-15中际旭创9.48%加仓被vectorbt以`NoCashLong`拒绝；
- 2026-05-18拓荆科技加仓只能部分成交。

最终收益在收盘代理方案中偶然接近，不代表逐日持仓一致；6月22日两套净值一度相差3.536个百分点。这一诊断随后促成了下方的真实现金账本修复。

## 原始main、当前策略和vectorbt的关系

三者不是三套选股策略：

1. 原始`main`提交`a309f85`运行旧版买卖规则；
2. 当前策略引擎运行结构锚点和50%/62.5%/75%等新规则；
3. vectorbt不重新生成信号，只对第2层已经产生的订单做现金约束重放。

在同一份截至2026-07-10的冻结数据和100万元初始资金下：

| 层次 | 收益 | 最终名义持仓/实际持仓 | 隐含或实际现金 |
|---|---:|---|---:|
| 原始main | 16.181% | 均胜电子12%、瑞芯微16%、海光信息12.8% | 710,700元 |
| 当前结构策略，收盘代理 | 41.921% | 均胜6%、拓荆51.2%、德明利18%、中际22.76% | -9,010元（名义核算） |
| vectorbt重放收盘代理 | 41.949% | 按实际成交数量盯市，另有伟测残余仓 | 158,465.85元 |
| 当前结构策略，次日开盘 | 34.360% | 均胜6%、拓荆57.6%、伟测12.8%、中际22.76% | -51,550元（名义核算） |
| vectorbt重放次日开盘 | 33.863% | 三次现金约束部分成交后的实际数量 | 11,047.78元 |

原始main与当前策略最终股票不同，主要来自结构信号、入场和退出规则变化，与vectorbt框架无关。当前策略与vectorbt最终数量不同，才来自现金不足、部分成交，以及原引擎使用名义成本/合并平均成本计算卖出数量的近似。

收盘代理方案的主要最终数量差为：

- 拓荆科技：原模型980.885股，vectorbt 950.030股；
- 中际旭创：原模型313.164股，vectorbt 181.144股；
- 伟测科技：原模型应为空仓，vectorbt仍余15.778股。

伟测残余仓说明问题不仅是“钱不够”：原引擎以名义仓位和合并平均成本计算退出收益，没有始终按真实批次股数扣减。后续真实现金化时应同时改造买入股数、逐批可卖股数和卖出扣减，不能只在最后加一个现金上限。

## 尚未覆盖

这层重放不宣称解决：

- 日K无法判断同一根K线内多个价位的先后顺序；
- T+1可卖数量；
- 停牌、涨跌停封单是否真实可成交；
- 分批持仓使用算术平均成本带来的份额近似；
- 历史候选和财务数据本身的时点完整性。

这些问题应分别在策略撮合层和数据层处理，不能用两套账户结果接近来替代验证。

## 真实现金账本修复结果

现有策略引擎现已改为：

- 每次买入先从共享现金中预留万0.85、最低5元和滑点；
- 现金不足时在策略引擎内部缩量，不再产生隐性融资；
- 每个交易批次保存实际成交股数和真实成交成本；
- 卖出按批次实际股数扣减，合并止盈使用股数加权成本；
- 日终净值直接使用`现金 + 持仓股数 × 收盘价`；
- 事件同时保留展示价和未取整`execution_price`供独立重放。

同一份截至2026-07-10的冻结数据、100万元初始资金修复后：

|---|---:|---:|---:|---:|---:|
| 14:55/收盘代理 | 42.409% | 42.409% | -17.061% | 0 / 0 | 低于0.000001pct |
| 收盘确认后次日开盘 | 35.768% | 35.768% | -14.138% | 0 / 0 | 低于0.000001pct |

收盘代理方案有7笔计划订单在当前引擎内因现金不足主动缩量，次日开盘方案有6笔；它们不再以借款或事后拒单的方式进入持仓。两套引擎最终逐股数量一致到约`1e-8`股，异常残仓已经消失。该段是整手规则接入前的历史结果。
